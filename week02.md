## Chapter 3 — 첫 번째 배포 파이프라인

### Push 배포의 문제는 결국 드리프트

`kubectl scale`, `kubectl set image` 같은 명령형 조작은 빠르지만, Git과 클러스터가 어긋나기 시작하는 순간이다. 회사에서 운영하며 실제로 겪는 문제이기도 하다 — 급해서 손으로 고친 변경이 이력 없이 남고, 며칠 지나면 아무도 이유를 모른다. 직전 상태의 기록이 없으니 롤백 기준도 없다.

GitOps는 이걸 구조로 막는다. **Git에 적힌 것이 곧 클러스터의 상태**이고, 배포는 사람이 클러스터에 미는(push) 게 아니라 ArgoCD가 Git을 보고 끌어오는(pull) 방식이 된다. 모든 변경은 커밋으로 남고, 롤백은 `git revert`다.

### ArgoCD 연결

Application을 `automated` + `prune` + `selfHeal`로 선언하는 게 핵심이다. 특히 selfHeal — 손으로 replicas를 바꿔도 몇 초 뒤 Git 값으로 되돌아온다. 드리프트를 "감시"하는 게 아니라 **성립 자체를 못 하게** 만드는 구조라는 점이 마음에 든다.

설치 자체는 공식 매니페스트인데, 저자 가드레일의 트러블슈팅에 미리 적혀 있는 주의점 두 개는 기억해둘 만하다: CRD가 커서 `--server-side` apply가 필요하다는 것, v3부터 NetworkPolicy가 기본 포함이라 repo-server의 GitHub 접근이 막혀 "sync는 됐다는데 반영이 안 되는" 상태가 나올 수 있다는 것. 실전에서 시간 잡아먹는 유형의 문제들이라, 이런 게 문서화되어 있는 게 이 책 실습 자료의 장점이다.

### git push만으로 배포, 그리고 롤백

버전 확인용 엔드포인트(`/version`)를 추가하고, 매니페스트의 이미지 태그만 바꿔 push하면 ArgoCD가 감지해 롤링 업데이트한다. `maxSurge: 1` / `maxUnavailable: 0`이면 새 Pod이 먼저 뜨고 나서 옛 Pod이 내려간다.

롤백은 `git revert HEAD && git push`. `kubectl rollout undo`와의 결정적 차이는 **되돌림 자체가 커밋으로 남아서 "왜 되돌렸는지"까지 추적된다**는 점이다.

### GitHub Actions CI — 여기가 EKS 치환의 중심

책은 GCP 서비스 계정 인증으로 Artifact Registry에 이미지를 push한다. 내 환경으로 옮기면:

| 책 (GCP) | 내 환경 (AWS/EKS) |
|---|---|
| Artifact Registry | ECR |
| SA 키 JSON을 GitHub Secrets에 저장 | **GitHub OIDC + IAM Role** (장기 키를 저장소에 안 둠) |
| `google-github-actions/auth` | `aws-actions/configure-aws-credentials` |
| `gcloud auth configure-docker` | `aws-actions/amazon-ecr-login` |
| `gcloud builds submit` (Cloud Build) | runner에서 `docker build` (로컬 빌드 시엔 Apple Silicon이라 `--platform linux/amd64` 주의) |

인증은 정적 키를 GitHub에 넣는 대신 OIDC로 가기로 했다. 실무에서도 이 방향으로 넘어가는 추세라 겸사겸사 손에 익혀둘 생각이다.

### CI와 ArgoCD 잇기

CI가 빌드까지 해도, 새 이미지 태그를 매니페스트에 반영하는 주체가 없으면 결국 사람이 태그를 고치게 된다. 그래서 CI 마지막에 매니페스트의 태그를 갱신해 커밋·push하는 스텝을 붙이고, ArgoCD가 그 커밋을 배포하게 한다.

다만 자동 커밋이 다시 CI를 트리거하는 루프가 될 수 있어 보인다. 빌드 트리거를 `app/` 경로 변경으로 한정해 매니페스트 커밋에는 반응하지 않게 끊는 게 기본 방어인데, 실습 때 실제로 루프가 안 도는지 확인해볼 항목으로 적어뒀다.

### 마무리 — CLAUDE.md에 배포 규칙 넣기

"클러스터 직접 변경(kubectl apply/delete)은 거부하고 Git 경유를 제안하라"는 행동 규칙을 CLAUDE.md에 넣고, 일부러 "지워줘"라고 시켜서 거부하는지 확인하며 장이 끝난다. 그런데 이 대목에서 걸리는 게 있어서 소감에 따로 적었다.

## Chapter 4 — 관측 가능성 한 번에 구축하기

### 세 기둥

메트릭(**무엇이** 일어나는가) / 로그(**왜** 일어났는가) / 추적(**어디서** 느린가). 이 장은 앞의 둘에 알림까지를 다루고, 추적은 뒤에서 다룬다고 한다.

### 메트릭: kube-prometheus-stack

Prometheus·Grafana·Alertmanager·exporter를 Helm 차트 하나로 설치한다. 내 노드가 작아서(t3.medium×2) requests를 줄인 values로 설치하는 게 전제다.

EKS라서 미리 알고 들어가는 것 하나 — 관리형 클러스터라 컨트롤플레인 컴포넌트(kube-controller-manager, kube-scheduler, etcd)는 스크레이프가 안 된다. 해당 타깃의 down은 **정상**이다. 이걸 모르고 보면 설치가 잘못된 걸로 오해하기 딱 좋은 지점.

### 로그: Loki + Fluent Bit

Loki는 로그 본문이 아니라 **라벨만 인덱싱**해서 가볍다. 이 클러스터 규모에 ELK를 올리는 건 메모리부터 무리라 선택의 여지가 별로 없다. 가드레일 트러블슈팅을 보면 최신 차트에서 schema 설정(`useTestSchema`)이나 스토리지 정의를 빠뜨리면 기동 자체를 못 한다고 하니, values 파일을 미리 만들어두고 시작하는 게 좋겠다.

### 알림: PrometheusRule

Pod 재시작 과다 같은 알림 조건을 CRD로 Git에 선언한다. 가드레일에 굵게 경고된 함정 하나: **PrometheusRule의 `release` 라벨이 Helm 릴리스 이름과 일치해야** Operator가 규칙을 읽는다. "규칙 만들었는데 안 뜬다"의 원인이 될 것 같아 적어둔다.

### 마무리 — 맥락 기록

작업의 결과는 Git에 남지만, "왜 이 값으로 했는지, 뭘 미뤘는지" 같은 맥락은 세션이 끝나면 사라진다. 항상 로드되는 규칙은 CLAUDE.md에, 작업 맥락·결정·TODO는 memory에 나눠 남기라는 내용. 세션을 이어 쓰는 도구라서 나오는 운영 습관이다.

## 소감

- 책이 GKE 기준이라 EKS로 옮기는 작업이 클 줄 알았는데, 막상 정리해보니 클라우드 종속 구간은 CI 인증부 정도로 국소적이었다. ArgoCD나 관측 스택처럼 도구 대부분이 클라우드 중립이라는 걸 확인한 게 이번 주의 수확.
- 다만 가드레일 체계에는 의문이 하나 남는다. 최근 고객사의 에이전트 구성 작업을 하고 있는데, 프롬프트만으로는 백퍼센트 동작을 장담하지 못하는 경우가 많아 서버 로직 상에서 강제하게 되는 경우가 많아지고 있다. 이 책의 가이드는 전부 **마크다운 문서**다. 실행 절차도, "kubectl delete를 거부하라"는 행동 규칙도 다 자연어라서, LLM이 따르도록 유도할 뿐 강제하지는 못한다. 책의 기준 실행 방식이 `--dangerously-skip-permissions`라는 점까지 겹치면 더 걸린다. LLM은 확률적으로 동작하니 100% 준수는 보장되지 않는데, 문서야 틀리면 고치면 되지만 인프라는 `kubectl delete` 한 번이 사고다. 최근에 쿠베 환경을 LLM 으로 관리하는 중 파드 재시작을 의도하고(디플로이가 있는 경우 파드를 지우면 재시작하니까) "파드 지워줘"라고 했는데 LLM이 디플로이 자체를 삭제한 경우가 있었다. 이렇게 지침이 명확하지 않거나 혼동을 줄 수 있는 경우 식은땀 흐르는 일이 생길 수도 있을 것 같다. **99번 잘 따르다가 1번 어기는 시스템을 프로덕션에 둘 수는 없다.**
- 그래서 프롬프트 바깥의 강제 장치 — 도구 설정에서 파괴적 명령 차단, 세션에 물리는 kubeconfig 권한 축소, 변경은 CI 경유로만 허용 같은 것들 — 를 이런저런 식으로 고민만 해본 상태다.

결국 "AI를 믿는다"가 아니라 **"어겨도 시스템이 막아준다"**가 되어야 안심하고 맡길 수 있다. 자연어 가드레일은 1차 필터로는 훌륭하지만, 그것만 믿고 승인 생략 모드로 인프라를 만지는 건 학습 환경에서나 할 일이라고 본다. 이 우려가 실제로 어느 정도 문제가 되는지, 남은 장들을 실습하면서 계속 확인해볼 생각이다.
