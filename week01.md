> 기간: 2026.06.29 ~ 07.05

## 들어가며

업무에서 EKS 기반 쿠버네티스를 운영하고 있어서 1장의 문제의식 자체는 익숙했다. 대신 이번 스터디에서 얻고 싶은 건 두 가지다.

1. GitOps까지는 익숙한데, 그 위에 AI를 얹은 **GitAIOps**라는 운영 방식이 실제로 어디까지 동작하는지
2. 책이 제시하는 **가드레일 / 결과 템플릿 문서 구조**를 내 환경 기준으로 재작성해보는 것

실습은 책의 GKE 대신 **AWS(EKS)** 로 진행한다. 평소 쓰는 스택이 AWS라 학습 전이가 빠르고, 책의 의도가 "특정 클라우드"가 아니라 "선언과 자동화의 구조"라서 클라우드를 바꿔도 무방하다고 판단했다. (채널에서도 Azure, 홈서버 k3s 등 각자 환경으로 변형해서 진행하는 분위기)

## Chapter 1 — AI 시대, 개발자의 인프라

### 개발자에게 인프라가 온 이유

- 2010년 이전엔 개발/운영 조직이 분리되어 있었지만, DevOps 확산 이후 "you build it, you run it" — 배포, 장애 대응, 파이프라인이 개발자 책임 범위로 들어왔다.
- IaC가 표준이 되면서 인프라도 결국 코드(Git)로 관리된다. Terraform·YAML·Git 파이프라인 조합이 사실상 기본값.

### Kubernetes = 클라우드의 공통 언어

- AWS/GCP/Azure 어디서든 같은 방식으로 배포·네트워킹·스토리지·보안을 추상화한다. "클라우드를 위한 운영체제"라는 표현이 적절.
- 회사가 클라우드를 옮겨도 k8s 위에서의 운영 지식은 그대로 이식된다 — 학습 투자 대비 수명이 긴 이유.

### GitOps → GitAIOps

Git을 단일 진실 공급원(SSOT)으로 두고 ArgoCD/Flux가 동기화하는 GitOps 위에, AI가 "작성·진단·검증·문서화"를 얹는다.

| 영역 | GitOps | GitAIOps |
|---|---|---|
| 상태 정의 | 사람이 YAML 작성 | AI가 요구사항에서 manifest 생성 |
| 트러블슈팅 | 사람이 로그·이벤트 분석 | AI가 로그·메트릭 분석 후 원인·조치안 제시 |
| 검증 | 수동 확인 | drift 감지·검증 자동화 |
| 문서화 | 별도 관리 (대개 뒤처짐) | 작업과 함께 자동 누적 |

한 줄 요약: **자연어로 지시하면 Git에 선언되고, 클러스터에 자동 반영된다.** Git은 신뢰성·추적성을, AI는 작성·진단의 효율을, Ops는 실행의 안전성을 담당한다.

### 이 장의 진짜 핵심 — CLAUDE.md와 가드레일 (1.6)

AI에게 인프라 작업을 맡길 수 있는 이유는 모델이 똑똑해서가 아니라 **행동 범위를 문서로 제약**하기 때문이다.

- `CLAUDE.md` — 매 세션 AI가 참조하는 프로젝트 컨텍스트
- 가드레일 문서 — 입력을 탐색/비교/실행으로 분류하고, kubectl 컨텍스트를 강제하며, 작업 전후 검증 절차를 명시
- `JOURNEY.md` — 진행 상황·버전·리소스 현황을 AI가 갱신하며 기록
- 실습 구조는 저자 제공 `_book-gitaiops`(가드레일·의사결정 가이드) + `notiflex-platform`(실습 앱) 두 저장소. 가상 스타트업 Notiflex의 창업(2장) → 성장 → 운영 표준화(9장) 타임라인을 따라간다.

## Chapter 2 — 실습 환경 구성 (책: GKE → 나: EKS)

### GKE ↔ AWS 매핑

| 책 (GCP) | 내 환경 (AWS) |
|---|---|
| gcloud CLI + 프로젝트 | aws CLI + 스터디 전용 계정 프로파일 |
| GKE Standard Zonal (e2-medium ×2, Spot VM) | EKS + managed node group (t3.medium ×2, Spot) |
| Artifact Registry | ECR |
| Gateway API 옵션 활성화 | Gateway API는 별도 컨트롤러 설치 필요 (5장 전 준비) |
| asia-northeast3 (서울) | ap-northeast-2 (서울) |

### 진행 순서

1. 도구 준비 — Claude Code(터미널), aws CLI, eksctl, kubectl 설치·버전 확인
2. **스터디 전용 AWS 계정에 프로파일 분리** — 업무 계정과 절대 섞이지 않게. 가드레일에도 "작업 전 계정/컨텍스트 확인"을 1순위로 명시했다.
3. eksctl로 클러스터 생성:

```yaml
# cluster.yaml (요지)
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: gitaiops
  region: ap-northeast-2
managedNodeGroups:
  - name: ng-spot
    instanceTypes: ["t3.medium"]
    desiredCapacity: 2
    spot: true
    volumeSize: 30
```

4. `kubectl get nodes`로 2노드 Ready 확인, kubeconfig 컨텍스트 이름을 명시적으로 변경 (가드레일의 "컨텍스트 강제" 적용)
5. notiflex-api 이미지를 ECR에 push → manifest 배포 → `kubectl port-forward`로 `/health`, `/id` 응답 확인

### AWS로 진행하며 주의한 점

- **비용 구조가 GKE와 다르다.** GKE는 월 $74 관리비 크레딧이 있어 실습 부담이 거의 없지만 (채널 실측 공유 기준 2~3시간 실습에 몇백 원 수준), **EKS는 컨트롤플레인 $0.10/h가 무조건 과금**되고 노드 EC2 비용이 별도다. → 실습 단위로 생성하고 끝나면 `eksctl delete cluster`로 즉시 삭제하는 사이클로 진행. 다음 장부터는 생성~삭제를 스크립트로 만들어둘 생각.
- Spot 노드그룹이면 책의 Spot VM 대응으로 충분하다. 실습용이라 중단 리스크는 무시 가능.
- Gateway API가 GKE처럼 옵션 한 줄로 켜지지 않는다. 5장(무중단 배포) 전에 컨트롤러 설치를 미리 검증해둘 것.
- 로컬 kubeconfig·자격증명 파일 덮어쓰기 주의 — 실습 전 백업 (채널에서 공유된 팁).
